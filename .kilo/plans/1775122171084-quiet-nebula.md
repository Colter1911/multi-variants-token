# Plan: manual token alpha + custom frame fixes

## Goal

Bring manual token creation in line with the requested behavior:

1. The locked centering circle must no longer behave as pre-existing alpha for custom frames.
2. Only user-drawn alpha should affect subject-over-frame behavior; additive alpha may extend outside the circle, and subtractive alpha may cut holes anywhere.
3. Subtractive alpha must remain visualized with red fill in the editor.
4. Final created tokens must match the preview when a custom frame or alpha extends beyond the base square, without shrinking the apparent token scale.
5. When a custom frame is active, preview scale must not be clamped to `1`; the chosen composition scale is authoritative.
6. Anything that extends beyond the preview square must remain present in the exported token image and must not be clipped or auto-fit back into the square.

## Findings

- The overflow/preview-scale fixes are now behaving as intended and should be preserved.
- The remaining unwanted behavior is the earlier change that made the default locked circle act like subject alpha.
- The base circle should return to being only the selection guide; it should not automatically count as user alpha in either compositing or editor fill visualization.
- User-drawn additive/subtractive alpha behavior should remain intact.
- The current custom-frame flow conflates two different concepts:
  - composition scale, which should define the real token composition
  - preview viewport size, which is only the visible square in the UI
- Preview and final export use different canvas behavior for custom frames:
  - preview visually lives inside a fixed square viewport
  - export can and should grow the real canvas to preserve overflow content
- Persisted custom-frame images are forced to `scaleX = scaleY = 1`, so a larger exported canvas gets fit back into the same token footprint and the visible subject appears smaller.
- The current preview zoom clamp for custom frames is logically wrong for the requested behavior, because it changes user intent instead of preserving the chosen composition and expanding the real export canvas.

## Implementation plan

### 1. Remove base-circle alpha from custom-frame compositing

Update `scripts/logic/AutoTokenService.mjs` in `_renderTokenCanvasFromCrop` so only explicit user alpha affects subject-over-frame compositing.

- Stop treating the base circle as part of the alpha mask used to keep the subject above the custom frame.
- Keep user additive polygons as the only regions that can lift the subject above the frame.
- Keep subtractive polygons able to cut holes anywhere.
- Preserve the overflow-canvas and texture-scale logic added later.

Expected result:

- the default locked circle is no longer treated as alpha
- additive alpha outside the circle still appears above the frame
- subtractive alpha still cuts holes anywhere and allows the frame/background to show through there

### 2. Remove base-circle fill from the stage alpha overlay

Update `scripts/apps/MultiTokenArtManager.mjs` in `#renderManualTokenStage`.

- Remove the cyan fill that currently appears for the locked base circle.
- Keep alpha overlay visualization only for explicit user operations:
  - additive polygons shown in cyan
  - subtractive cut areas shown in translucent red
  - keep the current red stroke feedback for an in-progress subtractive stroke
- Reuse the existing entry normalization so applied, pending, and current strokes remain consistent.

Expected result:

- the base circle goes back to being just the selection guide
- subtractive edits remain readable immediately via red fill

### 3. Unify preview and export render paths for overflow content

Update `scripts/logic/AutoTokenService.mjs` and `scripts/apps/MultiTokenArtManager.mjs` so preview and final creation use the same overflow-aware composition rules for custom frames.

- Extend `createTokenCanvasFromSelection` to accept `allowOverflowCanvas` and thread it into `_renderTokenCanvasFromCrop`.
- Make `_renderTokenCanvasFromCrop` return render metadata alongside the canvas, at minimum:
  - `baseCanvasSize`
  - `finalCanvasSize`
  - `textureScale` (derived from `finalCanvasSize / baseCanvasSize`)
- Remove the custom-frame preview clamp to `<= 1` in the manual dialog helpers.
- Treat the preview square as a viewport only: the chosen composition scale may push content outside that square, but the underlying render/export canvas must preserve the full bounds.
- Update `#renderManualTokenPreview` to use `allowOverflowCanvas: true` when a custom frame is active, so the preview is based on the same composition bounds as export.

Expected result:

- preview and export use the same geometry for overflowing frame/alpha content
- no more mismatch where export adds transparent padding that preview never showed
- a larger-than-square composition remains larger-than-square in the final asset instead of being clamped or re-fit

### 4. Preserve on-canvas scale after exporting an overflow canvas

Update `scripts/apps/MultiTokenArtManager.mjs` persistence flow.

- Capture render metadata from `createTokenBlobFromSelection` in `onCreate`.
- Pass the computed texture scale into `#persistGeneratedTokenBlob`.
- Store `scaleX` and `scaleY` using that computed scale for custom-frame outputs instead of hardcoding `1`.
- Apply the same rule both when creating a new token image entry and when replacing an existing token image.
- Update `#buildGeneratedTokenImage` so custom-frame defaults can use the computed scale value.

Expected result:

- the created token keeps the same apparent subject scale as preview
- overflowing decorative/frame elements extend outside the token footprint instead of forcing the whole artwork to shrink into the square
- the user-selected custom-frame scale remains authoritative even when it exceeds the preview square

## Verification

After implementation, verify these manual scenarios:

1. Lock a circle, enable custom frame, do not draw any extra alpha:
   - the circle alone does not create subject-over-frame alpha
   - only explicit user alpha changes subject-over-frame visibility
2. Additive alpha outside the circle:
   - portrait remains above the frame in those added regions
3. Subtractive alpha inside the circle:
   - the cut area is shown red in the editor
   - the final token has a true hole there
4. Large overflowing custom frame:
   - preview and created token match visually
   - subject scale does not shrink after applying the token
   - content outside the preview square is still present in the exported token image
5. Locked circle without any drawn alpha:
   - the circle itself does not create cyan alpha fill in the editor
   - the circle itself does not automatically keep the portrait above the frame
6. Replace an existing token image with a custom-frame manual token:
   - stored scale is updated correctly there as well

## Notes

- No localization changes are required unless you decide to surface new UI text.
- The safest implementation is to minimize antialias seams by using the exact same mask canvas for frame punch-out and subject compositing.
